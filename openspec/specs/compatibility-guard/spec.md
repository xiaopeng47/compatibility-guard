# 兼容性看护采集规范

## 1. 目标

本文档定义服务对接第三方系统时，需在 JDK、依赖库、配置等变更场景下采集的算法、协议及环境信息。采集结果用于建立兼容性基线，支撑日常版本迭代中的自动 diff 与告警。

## 2. 背景

服务存在大量对接第三方 Kafka、SFTP、HTTPS 等场景。在 JDK 升级、Kafka Client 升级、Apache SSHD 升级、HTTP Client/Server 升级过程中，经常因算法、协议、证书、SASL、Kerberos、TLS 等变更导致与第三方服务不兼容，且问题多在上线后被动发现。

## 3. 总体方案

采用**两层验证**落地兼容性看护：

```
┌─────────────────────────────────────────────┐
│           兼容性看护测试套件                 │
├─────────────────────────────────────────────┤
│                                             │
│  第一层：静态算法/协议清单检查                 │
│  • 维护 compatibility-inventory.yaml        │
│  • 检查 JDK / Client / Server 是否仍支持依赖算法 │
│  • 无外部依赖，本地/CI 秒级运行               │
│                                             │
│  第二层：动态握手探测                         │
│  • 维护 baseline/*.json 期望协商结果          │
│  • 连接代表服务或真实三方完成握手              │
│  • 验证实际协商结果是否与基线一致              │
│                                             │
└─────────────────────────────────────────────┘
```

两层共同输出兼容性报告，由开发者决策处理，不阻塞 PR 合并。

## 5. 采集优先级

| 优先级 | 含义 | 触发时机 |
|--------|------|----------|
| P0 | 核心兼容性项，失败即阻断 | 每次 PR、每次 JDK/依赖升级 |
| P1 | 基线采集与失败排查项 | 初始基线、nightly、失败时 |
| P2 | 深度诊断与特殊场景项 | 疑难问题排查、安全审计 |

## 6. 兼容性清单

`compatibility-inventory.yaml` 记录系统依赖的算法/协议，用于第一层静态清单检查。

清单文件格式、扫描项、检查规则、各专题采集字段详见 `openspec/changes/compatibility-test-guard/design.md`。

## 7. 基线

`baseline/*.json` 记录每个第三方、每个环境的期望协商结果，用于第二层动态握手探测。

基线文件内容、来源、维护方式详见 `openspec/changes/compatibility-test-guard/design.md`。

## 8. 看护触发条件

当以下任意项发生变化时，应触发兼容性看护：

1. JDK 主/次版本升级
2. JDK 小版本升级（可能伴随 security baseline 变化）
3. Kafka Client 版本升级
4. Apache SSHD / JSch / sshj 版本升级
5. HTTP Client / HTTP Server 版本升级
6. Spring Boot / 依赖管理平台升级
7. BouncyCastle / 国密 provider 升级
8. `java.security` 配置变更
9. `krb5.conf` / JAAS 配置变更
10. keytab / 证书轮换
11. 容器基础镜像升级
12. `compatibility-inventory.yaml` 清单变更
13. `baseline/*.json` 基线变更
