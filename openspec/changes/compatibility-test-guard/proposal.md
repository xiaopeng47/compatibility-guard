# 兼容性看护测试套件

## 背景

服务存在大量对接第三方 Kafka、SFTP、HTTPS 的场景。在 JDK 升级、Kafka Client 升级、Apache SSHD 升级、HTTP Client 升级过程中，经常因算法、协议、证书、SASL、Kerberos、TLS 等变更导致与第三方服务不兼容，且问题多在上线后被动发现。

## 目标

将兼容性验证从上线后发现左移到日常开发阶段，通过可运行、可维护的测试套件提前暴露风险。

## 范围

### 包含

- 第三方 Kafka 传输层兼容性（TLS、SASL、证书）
- 第三方 Kafka Kerberos 认证兼容性（高优先级专题）
- 第三方 SFTP/SSH 兼容性（KEX、cipher、MAC、host key、认证）
- 第三方 HTTPS 入站兼容性（我方 HTTPS 服务端与第三方客户端的 TLS 协商）
- 第三方 HTTPS 出站兼容性（我方 HTTPS 客户端与第三方服务端的 TLS 协商）
- JDK 算法支持性基线验证

### 不包含

- PR 合并门禁（不拦截合并流程）
- 独立看护平台或 Diff 引擎系统
- 生产环境监控或告警系统

## 方案概述

采用**两层防线**落地兼容性看护：

```
┌─────────────────────────────────────────────┐
│           兼容性看护测试套件                 │
├─────────────────────────────────────────────┤
│                                             │
│  第一层：静态算法/协议清单检查                 │
│  • 维护 compatibility-inventory.yaml        │
│  • 检查 JDK / Client / Server 是否仍支持依赖算法 │
│  • 无外部依赖，本地/CI 秒级运行               │
│                                             │
│  第二层：动态握手探测                         │
│  • 维护 baseline/*.json 期望协商结果          │
│  • 连接代表服务或真实三方完成握手              │
│  • 验证实际协商结果是否与基线一致              │
│                                             │
└─────────────────────────────────────────────┘
```

以工程化测试套件形式落地：

- 提供按专题组织的测试用例
- 提供可复用的探测辅助工具（清单扫描器 + 握手探测器）
- 使用清单文件记录依赖的算法/协议
- 使用基线文件保存各三方期望协商结果
- 默认使用代表服务运行，扩展支持三方真实环境
- 在本地、CI、nightly 均可运行
- 测试失败仅暴露问题，不阻塞流程

## 预期收益

| 场景 | 预期效果 |
|------|---------|
| JDK 17 升级 | 清单检查直接发现 cipher / Kerberos etype 被禁用；握手探测验证协商结果 |
| Kafka Client 升级 | 清单检查显示默认算法列表变化；握手探测显示 SASL / hostname verification 行为变化 |
| Apache SSHD 升级 | 清单检查显示支持算法变化；握手探测显示 KEX / cipher 协商变化 |
| HTTP Client / Server 升级 | 清单检查显示 TLS / cipher / ALPN 支持变化；握手探测验证 HTTPS 入站/出站协商 |
| 证书轮换 | 握手探测显示 fingerprint / 有效期变化 |
| 日常配置变更 | 清单 + 握手探测共同验证兼容性 |

## 关键设计决策

1. **不做 PR 门禁**：测试失败不拦截合并，由开发者决策处理。
2. **两层验证**：静态清单检查保证“必要条件”，动态握手探测验证“充分条件”。
3. **测试套件形态**：不建独立平台，以工程化测试落地。
4. **Kerberos 独立专题**：因依赖 KDC 和 JDK 强封装，问题模式独特。
5. **HTTPS 双向看护**：同时覆盖入站（第三方调我方）和出站（我方调第三方）场景。
6. **代表服务优先**：日常测试用容器 / 本地服务，稳定可控。
7. **清单与基线随代码管理**：依赖清单和基线数据作为测试资源，和代码一起版本化。
8. **三层探测**：清单检查 + 集成测试 + E2E 测试。
