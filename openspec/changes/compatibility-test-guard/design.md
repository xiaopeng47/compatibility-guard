## 2. 看护专题

### 2.1 Kafka Transport 专题

关注：
- TLS 版本
- Cipher suite
- SASL 机制
- 证书信息（SAN、issuer、有效期、fingerprint）
- Kafka API 版本
- 连接行为参数

### 2.2 Kafka Kerberos 专题（高优先级）

关注：
- JAAS / keytab / principal
- Kerberos etype 协商
- KDC / realm 配置
- Ticket 生命周期与续期
- JDK 17+ 强封装 JVM exports

### 2.3 SFTP/SSH 专题

关注：
- KEX 算法
- Host key 算法与 fingerprint
- Ciphers（client-to-server / server-to-client）
- MACs（client-to-server / server-to-client）
- 认证方式 / 公钥算法
- SFTP 协议版本

### 2.4 JDK Security 基线

关注：
- JDK 版本
- `java.security` 禁用算法
- Security providers
- Kerberos permitted enctypes

## 3. 测试分层

```
                    ┌─────────────────┐
                    │    E2E 测试      │
                    │  连接三方真实环境 │
                    │  慢、真实、手动跑 │
                    └────────┬────────┘
                             │
            ┌────────────────┴────────────────┐
            │         集成测试（主力）          │
            │  使用代表服务验证 handshake       │
            │  稳定、可重复、CI 可跑           │
            └────────────────┬────────────────┘
                             │
            ┌────────────────┴────────────────┐
            │           单元测试               │
            │  验证 JDK 算法支持性、配置解析    │
            │  快、无外部依赖                  │
            └─────────────────────────────────┘
```

| 层级 | 作用 | 速度 | 稳定性 | 外部依赖 |
|------|------|------|--------|---------|
| 单元测试 | 验证 JDK 算法支持性、配置解析 | 快 | 高 | 无 |
| 集成测试 | 验证 handshake 协商结果 | 中 | 高 | Testcontainers 代表服务 |
| E2E 测试 | 验证三方真实环境 | 慢 | 中 | 三方测试环境 |

## 4. 组件职责

| 组件 | 职责 |
|------|------|
| 测试用例 | 按专题组织，定义“测什么”、“期望什么” |
| 探测辅助工具 | 封装握手探测逻辑，输出结构化结果 |
| 基线加载器 | 读取各三方基线数据 |
| 基线文件 | 存储每个三方、每个环境的期望协商结果 |
| 代表服务 | 模拟三方算法配置，提供稳定测试目标 |
| 测试报告 | 展示当前结果、基线、diff、适配建议 |

## 5. 基线管理

### 5.1 基线内容

每个三方、每个环境保存一份期望的协商结果。基线文件只存储需要对比的核心字段：

| 专题 | 基线字段 |
|------|---------|
| Kafka Transport | TLS 版本、cipher suite、SASL 机制、证书 fingerprint |
| Kafka Kerberos | principal、etype、ticket 生命周期、renewable、JVM exports |
| SFTP/SSH | KEX、host key 算法与 fingerprint、cipher、MAC、认证方式 |

### 5.2 基线来源

| 来源 | 优先级 | 说明 |
|------|--------|------|
| 三方测试环境 | 最高 | 最真实 |
| 家里代表服务 | 常用 | 稳定可控 |
| 生产环境只读采样 | 兜底 | 需授权 |

### 5.3 基线维护

- 新三方接入时采集一次
- 三方配置变更后更新
- 本方主动升级并验证后更新
- 定期刷新（建议每季度）

## 6. 数据流

1. 测试用例启动
2. 加载对应三方基线
3. 探测辅助工具连接代表服务或真实环境
4. 采集当前协商结果
5. 与基线进行对比
6. 输出测试结果与 diff
7. 失败时输出适配建议

## 7. CI 策略

| 场景 | 运行内容 | 失败影响 |
|------|---------|---------|
| PR CI | 跑集成测试 + 单元测试 | 不阻塞 PR，仅暴露问题 |
| 每日构建 | 跑全量集成测试 | 告警，人工跟进 |
| 发布前 | 跑 E2E + 全量矩阵测试 | 发布评审参考 |
| 本地开发 | 按需运行相关专题测试 | 即时反馈 |

## 8. 采集项使用映射

采集项从 `openspec/specs/compatibility-guard/spec.md` 引入，在测试套件各环节的使用方式：

| 使用环节 | 作用 |
|---------|------|
| 测试用例设计 | 每个采集项对应一个验证点 |
| 探测辅助工具 | 运行时采集当前环境实际值 |
| 基线文件 | 存储期望结果 |
| 断言 / diff | 判断当前值是否符合基线 |
| 适配建议 | 失败时匹配已知模式，推荐修复方法 |
| 测试报告 | 把人关心的字段和 diff 展示出来 |

## 9. 安全考虑

- keytab、证书私钥等敏感凭证使用 CI secret manager 或本地安全存储
- 测试代码中不硬编码敏感信息
- 生产环境采样需获得授权，使用只读方式
- 基线文件中不包含私钥
